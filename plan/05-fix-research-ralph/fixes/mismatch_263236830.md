# Fix: mismatch_263236830 - Global init with Paren expression lost

## Summary

Global variable initializers using parenthesized expressions like `(-3)` are silently ignored, causing the variable to be zero-initialized instead.

## Test Case

```c
static int8_t g_2 = (-3);

int main(void) {
    return (int)(g_2) & 127;  // Expected: 125, Actual: 0
}
```

## Root Cause

The `evaluateConstantInitializer` function in `pkg/clightgen/program.go` handles:
- `cabs.Constant` - direct constants like `5`
- `cabs.Unary` with `OpNeg` - negative constants like `-3`

But it does NOT handle `cabs.Paren`, which wraps the expression when parentheses are used.

When parsing `(-3)`, the AST is:
```
Paren{Expr: Unary{Op: OpNeg, Expr: Constant{Value: 3}}}
```

The function returns `nil` for unhandled cases, causing the global to have no `Init` bytes.

## Fix Plan

Modify `evaluateConstantInitializer` in `pkg/clightgen/program.go` to handle parenthesized expressions:

```go
func evaluateConstantInitializer(expr cabs.Expr, typ ctypes.Type) []byte {
    size := SizeofType(typ)
    switch e := expr.(type) {
    case cabs.Paren:
        // Recursively unwrap parenthesized expressions
        return evaluateConstantInitializer(e.Expr, typ)
    case cabs.Constant:
        // ... existing code
    case cabs.Unary:
        // ... existing code
    }
    return nil
}
```

## Related Issue

There's also a secondary issue where typedef names like `int8_t` are not resolved to their proper types. `TypeFromString` in `pkg/clightgen/types.go` returns `ctypes.Int()` for unrecognized types instead of handling standard integer typedefs.

## Test

After fix, both test cases should pass:
1. `static int8_t g_2 = (-3);` should initialize g_2 to -3
2. The resulting assembly should use `.byte 253` (or equivalent) instead of `.zero 4`
